# v0.1 External Review Prompt

Use this prompt with GPT 5.5 Pro / Very High or a human expert reviewer after providing the
[v0.1 Public Preview Review Packet](v0.1-review-packet.md) and linked documents.

```text
You are acting as an expert security/product consultant reviewing Ithildin v0.1 as a public-preview
candidate. Review from the supplied packet and linked docs. If you cannot inspect code directly,
state that limitation and distinguish documentation risk from implementation risk.

Project summary:
Ithildin is a local-first governed MCP/tool gateway for AI agents. It mediates local agent tool use
through trusted tool manifests, JSON Schema validation, local principal labels, deny-default policy,
approval, scoped execution, redaction, and tamper-evident local audit. v0.1 is explicitly a local
preview, not production security software.

Review objective:
Identify what should block a public v0.1 preview tag, what should be fixed before or immediately
after release, and what should be prioritized for v0.2. Do not brainstorm new powerful tool
capabilities unless they are framed as deferred risks.

Please review these areas:

1. Public v0.1 release blockers
   - What would you block before a public-preview tag?
   - Which blockers are must-fix versus wording/documentation fixes?
   - Which release checks or manual checks should be added?

2. Security boundary risks
   - Does the project correctly describe itself as a local mediation layer rather than a sandbox,
     EDR/MDM, SIEM, production identity system, hosted MCP platform, or compliance audit system?
   - Are host trust, local admin trust, MCP stdio trust, trusted manifests, trusted policies, and
     local audit limitations clearly stated?
   - Are any docs or feature names overclaiming the current security posture?

3. Approval and evidence binding
   - Review the patch apply approval model: stored proposal only, one-time approval, proposal hash,
     base file hash, manifest hash/version, policy evidence, principal, request hash, expiry, and
     atomic consume.
   - What approval equivalence, replay, race, policy drift, stale proposal, or review UX risks
     remain?
   - What evidence should be added to approval records, audit events, or UI review surfaces?

4. Filesystem and patch executor edge cases
   - Review workspace scoping, traversal blocking, hidden/sensitive path denial, `.git` internals,
     symlink/hardlink/race behavior, binary/encoding rejection, patch context validation, stale-base
     rejection, and atomic replace.
   - What additional tests or implementation hardening would you require before public preview?

5. HTTP fetch edge cases
   - Review exact allowlist matching, GET-only behavior, no caller headers, unsupported scheme and
     credential rejection, redirects, redirect-to-private, DNS rebinding/TOCTOU, IPv6, encoded IPs,
     IDNA/punycode, proxy environment variables, timeouts, body-size limits, and error leakage.
   - What SSRF or network edge cases remain highest risk?

6. Identity, policy, and audit
   - Local principals are attribution labels, not real authentication. Is that clear enough?
   - Are YAML policy and optional OPA evidence positioned safely?
   - Is hash-chained SQLite/JSONL audit correctly described as tamper-evident local evidence rather
     than immutable or notarized?
   - Should policy hashes, manifest hashes, and decision evidence be surfaced differently?

7. Product positioning
   - Is “local-first governed MCP/tool gateway” the right public label?
   - What phrasing should be avoided?
   - What must appear above the fold in README/release notes?

8. v0.2 priorities
   - Rank the next 6-10 tasks by trust impact.
   - Include must-do, high-value soon, and explicitly defer categories.
   - Assume no shell, Docker socket, Kubernetes, browser automation, arbitrary HTTP, broad writes,
     plugin marketplace, hosted MCP, production identity, runtime Postgres, or hosted telemetry
     should be added until core trust semantics are stronger.

Output format:

- Overall judgment: ready / ready with fixes / not ready.
- Release blockers: ordered list with severity and rationale.
- Public wording fixes: exact suggested language where useful.
- Technical hardening fixes: concrete tests or implementation changes.
- v0.2 priority roadmap: ordered tasks with why-now rationale.
- Do-not-add-yet list: dangerous expansions to keep deferred.
- Brutal short version: 5-8 sentences.
```
