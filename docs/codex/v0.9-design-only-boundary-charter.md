# v0.9 Design-Only Boundary Charter

v0.9 starts design-only capability planning under the v0.8 final decision packet. It may create
capability proposals, risk analyses, executor contracts, fixture sketches, review packets, and
validation gates. It must not implement new runtime behavior.

## Allowed Work

- Draft capability proposals and risk analyses.
- Define proposed executor contracts, policy fixtures, audit fields, UI evidence, negative cases,
  resource limits, and accepted-risk impact.
- Generate review packets for GPT 5.5 Pro / human review.
- Add validation scripts and tests that enforce design-only boundaries.

## Forbidden Work

No v0.9 task may add or change:

- tool manifests or `tool-manifests.lock.json`;
- executor code or governed tool dispatch;
- API or MCP runtime behavior;
- policy rules or approval behavior;
- production identity, runtime Postgres, hosted telemetry, or remote MCP;
- shell, Docker socket, Kubernetes, browser automation, arbitrary HTTP, broad filesystem writes, or
  plugin SDK work.

## Current Design Candidate

The first design-only candidate is `git.show.commit_metadata`. It is a proposal only. It does not
add a manifest, executor, policy rule, MCP exposure, or runtime behavior.

## Stop Conditions

Stop and report status if:

- a critical/high review finding appears;
- design wording implies implementation is approved;
- a tool manifest or runtime implementation surface changes;
- xhigh subagent and main-agent risk judgments materially disagree;
- the same gate fails three times;
- GPT 5.5 Pro / human external review is needed to resolve a product-risk ambiguity.

## Required Verification

- `make v09-design-only-gate`
- `make git-commit-metadata-proposal-check`
- `make git-commit-metadata-implementation-plan-check`
- `make v09-design-review-packet`
- `make release-check`
- `make review-candidate`

These commands do not approve implementation. They only verify that design-only planning stays
inside the v0.8 gate.
