# v0.6 Internal Proxy Review Operating Model

This document records the operating model for using high-intelligence subagents as a repeatable
internal proxy review bench while reserving GPT 5.5 Pro / Very High or a human expert for major
v0.X milestone consultation.

Internal proxy review is pressure testing. It is not external closure, not a substitute for
independent implementation review, and not permission to add new governed tool powers.

## Purpose

The goal is to reduce handoff latency while preserving Ithildin's safety posture:

1. automate routine source/evidence review;
2. catch implementation drift before a milestone packet leaves the repo;
3. turn findings into structured `SUB-###` records;
4. reserve external consultation for boundaries, high-risk findings, and v0.X go/no-go decisions.

## Lane Loop

For each focused lane:

1. Choose a narrow surface: patch apply, filesystem/platform, HTTP fetch, signed evidence, policy
   parity/registry, MCP ingress, review console/admin, or release automation.
2. Run the focused tests and packet/evidence command for that surface.
3. Spawn two or three high-intelligence subagents with disjoint scopes and source-file pointers.
4. Require each subagent to answer with judgment, evidence, residual risk, and structured findings.
5. Convert actionable results into `SUB-###` finding files.
6. Remediate high/medium findings or explicitly defer lower findings with accepted-risk rationale.
7. Run focused tests, then `make release-check`.
8. Update the closure matrix and lane notes as internal proxy reviewed, never externally closed.
9. Continue only when the tree is clean and the lane has no open critical/high internal finding.

## Stop Conditions

Stop autonomous implementation and report status when any of these happen:

- a critical or high internal finding appears;
- subagents disagree on a security boundary or product claim;
- the same focused test failure repeats after three fix attempts;
- a fix requires broadening the tool surface, MCP transport, identity model, storage backend,
  telemetry posture, or filesystem/network/write power;
- evidence cannot be made deterministic or secret-free;
- the repo cannot return to a clean verified checkpoint;
- the current issue is architectural rather than implementation-local.

## External Review Cadence

Use GPT 5.5 Pro / Very High or a human expert at these gates:

- after all v0.6 review lanes are internally proxy-clean;
- before any v0.X public-preview, broader positioning, or capability-expansion decision;
- before adding any new powerful tool class;
- after any critical/high internal finding, even if remediated;
- when a boundary, threat-model, identity, storage, telemetry, or MCP transport decision is unclear.

Purely mechanical docs/evidence cleanup does not need external consultation unless it changes claims.
