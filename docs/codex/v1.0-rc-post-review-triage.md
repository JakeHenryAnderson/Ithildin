# Ithildin v1.0 RC Post-Review Triage

Status: v1.0 local-preview post-review triage map.

This document describes how to process GPT 5.5 Pro / Very High or human expert feedback after the
v1.0 RC packet is reviewed. It does not add runtime behavior, tool manifests, executors, policy
rules, API/MCP behavior, UI runtime behavior, sandbox orchestration, SIEM adapters, production
identity, runtime Postgres, hosted telemetry, remote MCP, plugin SDK behavior, compliance
automation, or new governed tool powers.

## Intake Sequence

1. Save the raw reviewer response outside committed source until it is normalized.
2. Record the reviewed commit with `git rev-parse HEAD`.
3. Record the reviewed packet hash from `var/review-packets/v1.0/rc/v1-rc-artifact-hashes.json`.
4. Normalize actionable findings with `make external-response-normalize FILE=...`.
5. Convert each accepted actionable item into a reviewer finding using
   [Reviewer Finding Template](reviewer-finding-template.md).
6. Update [Source Review Closure Matrix](source-review-closure-matrix.md) only after triage.
7. Run `make reviewer-findings-check`, `make review-findings-summary`, and `make release-check`.

The external-response normalizer remains preparation only. It records `mutates_findings: false` and
`closes_external_review: false`; committed finding files and closure updates still require
manager-owned review.

Feature freeze remains active throughout post-review triage unless a separate post-RC decision
explicitly unfreezes a lane.

## Outcome Classes

| Reviewer outcome | Required action | Feature freeze effect |
| --- | --- | --- |
| No findings | Record a no-findings summary and keep pending rows visible unless source access is sufficient. | Freeze remains active. |
| Documentation-only should-fix | Patch docs under the feature freeze, run release gates, and keep capability expansion blocked. | Freeze remains active. |
| Packet/evidence gap | Add or refresh packet evidence only; do not change runtime behavior without a separate decision. | Freeze remains active. |
| Medium/low implementation finding | Create a finding, remediate in a scoped follow-up, and verify with focused plus release gates. | Freeze remains active. |
| Critical/high finding | Stop capability work, create a finding, keep RC handoff blocked until fixed or formally accepted-deferred. | Freeze remains active. |
| Product-boundary ambiguity | Stop and create a post-RC decision record before implementation or public-positioning changes. | Freeze remains active until decision. |
| Request for new powers | Reject as out of scope for v1.0 RC; require a later explicit capability or architecture sprint. | Freeze remains active. |

## Allowed Under Feature Freeze

- reviewer-response normalization;
- finding-file creation for accepted actionable issues;
- documentation or packet corrections that do not alter product claims;
- tests and gates that verify existing boundaries;
- source-review closure updates supported by reviewer evidence.

## Not Allowed Under Feature Freeze

- new manifests, executors, policy powers, MCP/API behavior, or UI runtime controls;
- sandbox orchestration, SIEM adapters, production identity, runtime Postgres, hosted telemetry,
  remote MCP, plugin SDK behavior, secrets-manager behavior, or managed model serving;
- public/security-product positioning;
- production/security/compliance positioning;
- closing external/source-review rows without sufficient reviewer access and explicit evidence.

## Required Commands

Use these commands after normalizing or applying review feedback:

```sh
make external-response-template-check
make external-response-normalize FILE=...
make reviewer-findings-check
make review-findings-summary
make v1-rc-post-review-triage-check
make release-check
make review-candidate
git status --short
```

For a no-findings packet-only response, normalization is still useful, but it must not close
source-review rows by itself.

## Stop Conditions

Stop and reassess before further implementation if:

- any critical/high finding is open;
- the same gate fails three times;
- a reviewer recommends changing the product boundary;
- a fix would require new runtime powers;
- a document would need to claim public/security-product or production readiness;
- a closure update would hide pending external rows.
