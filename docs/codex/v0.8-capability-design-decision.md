# v0.8 Capability-Design Decision

Wave 4 separates design-only capability planning from implementation.

## Decision Table

| Question | Decision | Conditions |
| --- | --- | --- |
| Capability design-only exploration | conditional_go | Proposals may be drafted only as design/risk artifacts. |
| Capability implementation | no_go | No manifests, executors, policy rules, MCP exposure, or runtime behavior changes. |
| New governed tool powers | no_go | Tool count remains 10 and no deferred power class is added. |
| High-risk capability classes | no_go | Shell, Docker socket, Kubernetes, browser automation, arbitrary HTTP, broad filesystem writes, secrets-manager tools, remote MCP, production identity, runtime Postgres, and hosted telemetry remain deferred. |

## Post-v0.9 Supersession Note

This v0.8 decision remains the historical design-only gate. The later
[v0.9 git.show.commit_metadata Implementation](v0.9-git-commit-metadata-implementation.md) record
supersedes the implementation block only for one bounded read-only Git metadata tool. It does not
approve broad capability implementation or any deferred power class.

## Required Proposal Evidence

Every future capability proposal must include:

- executor contract;
- policy fixtures and policy-preview expectations;
- audit fields and redaction behavior;
- UI/review evidence when an admin or approval flow is involved;
- negative transcripts;
- resource limits;
- accepted-risk impact analysis;
- no-new-powers boundary analysis;
- external/source review requirement before implementation.

## Constraints

Design-only work may compare options, define contracts, identify risks, and prepare external-review
questions. It must not add tool manifests, executor code, policy rules, MCP exposure, approval
behavior, production identity, storage backends, telemetry exporters, network transports, or broad
filesystem access.

`AR-010` constrains designs that might expose secrets, telemetry, or stronger redaction claims. Such
designs may be sketched only if they explicitly preserve best-effort redaction wording and require
external review before implementation.
