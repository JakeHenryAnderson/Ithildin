# v0.2 Review Response and RC Cleanup

This document summarizes the cleanup work completed after the v0.1 external review and defines the
final handoff posture for a v0.2 review candidate for the v0.1 local-preview runtime boundary. It is
meant to be read before [v0.2 Review Packet](v0.2-review-packet.md).

## Review Response Summary

The prior external review said Ithildin was credible as a local-preview governed MCP/tool gateway,
but should not be described as a production security product. It recommended hardening approval
binding, filesystem and HTTP edge cases, evidence visibility, local auth/deployment boundaries, and
release documentation before expanding tool powers.

The implemented response through Task 061 is:

- locally signed audit exports and manifest locks for local offline evidence checks;
- policy fixtures, policy impact preview, and shared decision evidence;
- named workspaces and stronger approval review surfaces;
- local admin-token setup guidance and dangerous-config status reporting;
- audit diagnostics, MCP Inspector recipes, redaction evidence, and evidence contracts;
- approval drift, path ambiguity, hardlink, Unicode, proxy, and canonical-host regression coverage;
- refreshed v0.2 roadmap, release-packet evidence, v0.2 review packet, release metadata, locally
  signed evidence demo fixtures, and negative review recipes.

## Release Candidate Cleanup Status

The current candidate is intentionally review-ready rather than feature-expanded. It preserves the
local-preview boundary:

- stdio MCP only;
- SQLite runtime storage only;
- local bearer-token admin API and review console;
- local principal labels, not managed identity;
- local tamper-evident audit, not hosted custody;
- exact-allowlist GET-only HTTP fetch;
- stored patch proposal plus approval-gated apply as the only write path.

No shell, Docker socket, Kubernetes, browser automation, arbitrary HTTP, broad filesystem writes,
plugin SDK, hosted MCP, production identity, runtime Postgres, or hosted telemetry has been added.

## Residual Risks To Review

These are the areas the external reviewer should still inspect closely:

- whether docs and UI labels avoid presenting mediation as OS isolation;
- whether signed local evidence is useful without overclaiming hosted trust;
- whether patch approval binding is precise enough for stored proposal apply;
- whether policy preview, fixture tests, impact preview, and runtime decision evidence stay aligned;
- whether filesystem path hardening covers practical local-preview risks;
- whether HTTP fetch normalization, redirects, DNS/IP checks, proxy behavior, and errors are robust;
- whether evidence fields are stable enough for users to build local review workflows around them.
