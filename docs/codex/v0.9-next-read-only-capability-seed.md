# v0.9 Next Read-Only Capability Seed

Status: candidate planning only.

This document records plausible next read-only capability candidates after
`git.show.commit_metadata`. It does not approve implementation, manifests, executors, policy rules,
MCP exposure, API behavior, UI behavior, or runtime changes.

The first follow-on proposal has been drafted as
[Capability Proposal: git.show.ref_summary](capability-proposals/git-show-ref-summary.md) and is
validated by `make git-ref-summary-proposal-check`. That proposal remains design-only and does not
authorize implementation.

## Selection Criteria

The next candidate should:

- stay read-only and local-only;
- use structured inputs, never caller-controlled argv or format strings;
- return bounded metadata, not file contents, raw diffs, raw patches, credentials, or shell output;
- preserve policy preview/runtime parity;
- preserve existing role visibility and audit evidence;
- include negative tests, resource limits, safe-error behavior, and a focused source-review bundle;
- pass xhigh review before implementation and after implementation.

## Preferred Candidate

### `git.show.ref_summary`

Potential behavior:

- return bounded metadata about local refs only;
- accept an optional structured selector such as `kind: branch|tag|all_local`;
- return ref name hashes or redacted names by default, resolved commit hashes, type, and counts;
- optionally include the current branch name only when explicitly requested and redacted/hashed in
  evidence;
- never include remote refs, reflogs, revision ranges, pathspecs, raw commit messages, file names,
  diffs, blob contents, checkout, branch mutation, remote fetch, credentials, or shell.

Why it is a reasonable next candidate:

- it complements `git.show.commit_metadata` without adding write power;
- it helps agents orient around local repository state without reading file contents;
- it can use fixed Git argv and bounded output;
- its privacy risks are visible and reviewable: branch/tag names can leak project intent, so default
  redaction or hashing should be part of the proposal.

Required proposal sections before implementation planning:

- exact input and output shapes;
- ref-name privacy policy;
- resource construction and policy/audit evidence;
- output limits and truncation behavior;
- denial cases for remote refs, revision syntax, pathspec-like selectors, control characters, and
  non-normalized names;
- MCP/UI expectations;
- accepted-risk impact;
- xhigh source-review plan and break-glass criteria.

## Other Candidates To Reconsider Later

- `git.show.tag_metadata`: likely safe if tag messages/signatures are excluded or tightly bounded,
  but annotated tag messages can contain untrusted text and possible secrets.
- `git.show.branch_summary`: useful but branch names are often sensitive; should probably be folded
  into `git.show.ref_summary`.
- `git.show.commit_file_summary`: higher risk because changed-file names can reveal sensitive
  repository structure; defer until path-name privacy policy is stronger.

## Explicit Non-Candidates For Now

- raw `git show`;
- raw `git diff`;
- blame output;
- arbitrary Git command execution;
- checkout, branch creation/deletion, merge, rebase, reset, stash, fetch, pull, push, or submodule
  mutation;
- any tool that returns blob contents or raw patch hunks.
