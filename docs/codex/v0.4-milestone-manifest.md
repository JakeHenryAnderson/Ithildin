# v0.4 Milestone Manifest

This manifest records the v0.4 review-closure and evidence-maturity roadmap through Task 151. v0.4
is not a new tool-power release. It is a hardening and review-closure wave over the same narrow v0.1
local-preview runtime boundary.

Tasks 113-122 are complete as the first v0.4 foundation and hardening goal. Tasks 123-151 are
planned only. They should not be treated as implementation-complete until each task has its own
checkpoint commit and gate output.

The machine-readable companion is
[v0.4-milestone-manifest.json](v0.4-milestone-manifest.json). That JSON file is the source of truth
for planned task IDs, wave groupings, gates, stop conditions, review points, and deferred
boundaries.

## Operating Rule

Each milestone must be independently committed. Each wave must pass its gate before the next wave
starts. Internal high-intelligence subagents may pressure-test implementation and evidence, but they
cannot close external source-review rows or approve new product-boundary claims.

Stop and report status if the same blocking failure repeats three times, if a trust-boundary
regression appears, if a critical/high internal finding appears, if the work requires changing the
product boundary, or if release/audit/manifest/signing evidence becomes ambiguous.

External GPT 5.5 Pro / Very High or human expert review is required before any new powerful tool
class, production/security-product positioning, remote MCP, production identity, runtime Postgres,
hosted telemetry, plugin SDK, shell/Docker/Kubernetes/browser tools, arbitrary HTTP, or broad
filesystem writes.

## Deferred Boundaries

The following remain deferred throughout v0.4:

- shell execution;
- Docker socket access;
- Kubernetes tools;
- browser automation;
- arbitrary HTTP methods, caller-supplied headers, request bodies, cookies, or broad network access;
- broad filesystem writes, deletes, moves, chmod, archive extraction, or secrets-manager tools;
- plugin SDK or marketplace;
- remote hosted MCP gateway;
- production identity integrations;
- runtime Postgres;
- hosted telemetry collectors;
- managed model serving or LLM proxy workflows.
