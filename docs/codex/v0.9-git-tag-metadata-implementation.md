# v0.9 git.show.tag_metadata Implementation Boundary

Status: approved v0.9 implementation boundary for one read-only Git tag metadata capability.

`git.show.tag_metadata` may proceed as a bounded continuation of the existing Git metadata lane. It
does not approve a new powerful tool class, arbitrary Git execution, raw tag exposure, Git trust
claims, or broader repository inspection.

## Approved Boundary

The implementation may:

- add one manifest: `tool-manifests/git-show-tag-metadata.yaml`;
- keep risk `read`, category `git`, and MCP exposure under the existing governed registry path;
- accept only `workspace_id`, `selector.kind: all_local_tags`, and bounded `limit`;
- use fixed argv Git subprocess calls controlled by Ithildin;
- query local `refs/tags` only;
- validate and suppress raw tag names before returning output;
- distinguish lightweight and annotated tags;
- return response-local `tag_id` handles, tag type, target object type, resolved commit hashes where
  safely available, count/truncation evidence, and skipped/error counts;
- skip non-commit tag targets safely;
- label all returned metadata as untrusted;
- preserve policy preview/runtime parity with a normalized `git_tags` resource;
- record audit metadata containing selector, count, truncation, skipped-count, and output-policy
  booleans only;
- update manifest lock, tool-surface/readiness gates, policy parity fixtures, MCP tests, and source
  review handoff material.

## Non-Goals

The tool provides no raw tag names by default, no stable tag-name hashes, no annotated tag messages,
no signatures, no file names, no file contents, no raw diffs, no remote refs, no remotes, no
refspecs, no pathspecs, no arbitrary revision selectors, no checkout, no tag creation/deletion, no
branch mutation, no remote fetch/pull/push, no credentials, no caller-controlled Git format strings,
no arbitrary Git argv, no shell, no package-manager execution, and no broad filesystem access.

## Privacy Caveat

`git.show.tag_metadata` remains local-preview metadata mediation, not Git provenance or release
trust evidence. Response-local `tag_id` values are display handles only. They are not stable
identifiers, privacy-preserving randomness, or cross-response non-correlation guarantees. The
privacy boundary is that raw tag names, stable tag-name hashes, tag messages, and tag signatures are
absent from responses and audit evidence.
