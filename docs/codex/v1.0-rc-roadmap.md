# Ithildin v1.0 RC Roadmap

Status: roadmap and sequencing target only.

This roadmap defines the remaining work toward a local-first Ithildin v1.0 release candidate. It
does not add runtime behavior, tool manifests, executors, policy rules, MCP exposure, API behavior,
UI behavior, sandbox orchestration, SIEM adapters, production identity, runtime Postgres, hosted
telemetry, remote MCP, plugin SDK behavior, compliance automation, or new governed tool powers.

## Target Definition

Ithildin v1.0 RC is a local-first governed MCP workbench for security-conscious developers. It
supports a small bounded read-only project/Git intelligence surface, approval-gated patch apply,
Agent Run evidence, local audit/export evidence, a usable local review console, repeatable local
demo workflows, and explicit local-preview/non-production boundary language.

v1.0 RC remains a local-preview release candidate. It is not production identity, not a hosted MCP
platform, not a kernel sandbox, not EDR/MDM/SIEM, not compliance automation, and not custody-grade
audit.

## Phase 1: Finish The Read-Only Metadata Surface

Goal: close the current bounded metadata family without adding a new power class.

Required checkpoints:

- keep `project.release.summary` inside the implemented bounded read-only metadata boundary;
- maintain governed pipeline, MCP, policy parity, audit metadata, negative transcript plans, and
  source-review bundle coverage;
- keep read-only capability inventory and next-capability readiness current;
- stop adding metadata tools unless a later v1.x proposal justifies them.

Done when:

- tool count reflects the implemented risk summary capability;
- `make release-check` and `make review-candidate` pass;
- the source-review bundle exists for the implemented lane;
- next-capability readiness reports no selected implementation candidate unless a new v1.x proposal
  is explicitly started.

## Phase 2: Prove The Local Workbench Loop

Goal: make the “agent in a governed local workbench” experience demonstrable end to end.

Required checkpoints:

- run a repeatable local MCP/client demo against a safe fixture workspace;
- record one mediated Agent Run with tool calls, approval evidence where applicable, audit evidence,
  and export evidence;
- make demo readiness, live demo packet, workbench packet, and operator walkthrough agree on the
  same reading order;
- keep all demo claims scoped to mediated actions through Ithildin.

Done when:

- `make demo-readiness-summary`, `make workbench-readiness`, `make demo-flow-readiness`,
  `make demo-evidence-readiness`, and `make review-candidate` pass;
- packet redaction scan reports no findings;
- the README gives a concise local demo path without implying production readiness.

## Phase 3: Polish The Operator Console

Goal: make the local review console feel like a coherent operator surface rather than scattered
diagnostics.

Required checkpoints:

- improve Agent Run timeline review, evidence export discoverability, warning states, and empty/error
  states;
- keep console actions read-only except existing approval/deny/export flows;
- verify UI tests cover trust warnings, run timeline states, export actions, approval binding
  evidence, and safe error rendering.

Done when:

- `npm run test --prefix apps/ui` and `npm run build --prefix apps/ui` pass;
- the console exposes no run controls, sandbox controls, broad write controls, or unreviewed
  mutation surfaces;
- UI wording preserves local-preview/non-production boundaries.

## Phase 4: Freeze Capability Expansion

Goal: stop feature growth and shift to release-candidate assurance.

Required checkpoints:

- declare a v1.0 RC feature freeze;
- reject new manifests, executors, policy powers, API/MCP behavior, and UI runtime surfaces unless
  they fix a blocking release issue;
- refresh no-new-powers, tool-surface invariant, policy parity, review findings, and release
  evidence gates;
- create a final v1.0 RC packet.

Done when:

- `make release-check` passes from a clean tree;
- `make review-candidate` passes from a clean tree;
- review packets clearly distinguish implemented local-preview behavior from deferred production
  features.

## Phase 5: Final Assurance And Handoff

Goal: hand a coherent v1.0 RC candidate to an external reviewer or trusted user without requiring
them to reconstruct the project history.

Required checkpoints:

- final source-review closure summary for implemented lanes;
- final accepted-risk/deferred-boundary summary;
- final install/run/demo/review instructions;
- final evidence packet with command transcripts and artifact hashes;
- final handoff prompt for GPT 5.5 Pro / human review if desired.

Done when:

- no critical/high findings remain open for the local-preview boundary;
- any medium/low findings have explicit disposition;
- capability expansion, production/security-product positioning, production identity, hosted trust,
  sandbox orchestration, SIEM custody, and compliance automation remain blocked or deferred.

## Delegation Rules For The Roadmap

- Main Codex owns scope, safety judgment, implementation review, gates, staging, and commits.
- Use one worker ticket at a time by default.
- Use low/mini workers for docs scans, stale wording checks, packet inventories, and narrow
  boilerplate suggestions.
- Keep runtime source, manifests, policy semantics, approval/audit behavior, MCP/API behavior, and
  product-risk decisions with main Codex or a higher-intelligence review path.
- Stop if a worker proposes boundary drift, if the same gate fails three times, or if implementation
  requires changing this roadmap’s deferred-power list.

## Deferred Beyond v1.0 RC

- production identity;
- runtime Postgres;
- hosted telemetry;
- remote MCP hosting;
- shell, Docker, Kubernetes, browser automation, arbitrary HTTP, or broad filesystem writes;
- sandbox orchestration;
- SIEM adapters or custody-grade audit;
- compliance automation;
- plugin SDK or marketplace;
- public/security-product positioning.
