# v0.9 git.show.commit_metadata Implementation

Status: approved v0.9 implementation for one read-only Git metadata capability.

`git.show.commit_metadata` adds bounded read-only Git metadata through the existing governed
pipeline. It is a narrow continuation of the already-reviewed Git read lane, not a new powerful
tool class.

## Boundary

This implementation:

- adds one manifest: `tool-manifests/git-show-commit-metadata.yaml`;
- keeps risk `read`, category `git`, and MCP exposure under the existing registry/listing path;
- uses fixed argv Git subprocess calls controlled by Ithildin;
- reads commit identity metadata with fixed per-field Git format calls rather than in-band
  separator parsing;
- accepts only structured ref selectors: full object ID, local branch, or local tag;
- fails closed unless the Git repository toplevel is within or equal to the configured workspace
  root;
- returns resolved commit hash, parent hashes, bounded subject/body metadata, optional email fields,
  timestamps, and redacted changed-file metadata;
- parses changed-file metadata from NUL-delimited Git output rather than raw diff hunks;
- redacts email values even when email metadata is requested, while still returning stable hashes;
- redacts email-like substrings embedded in author/committer names;
- labels returned subject/body metadata as untrusted and strips control/format characters;
- preserves policy preview/runtime parity with a syntactic `git_commit` resource;
- preserves existing policy rules, approval behavior, registry behavior, storage, identity, MCP
  transport, telemetry, and audit storage.

## Non-Goals

The tool provides no shell, no caller-supplied Git format strings, no arbitrary Git argv, no checkout,
no branch mutation, no remote fetch, no credentials, no file contents, no raw diffs, no blob contents,
no symlink targets, no broad filesystem access, and no new write path.

## Privacy Caveat

`git.show.commit_metadata` remains local-preview metadata mediation, not Git trust/provenance
evidence. Returned author/committer names, subject/body text, and changed-file names are
repository-supplied untrusted metadata. Email values are redacted and sensitive paths are hidden, but
stable email/path hashes are local evidence aids rather than anonymity guarantees; common values may
remain guessable by dictionary. Future stronger privacy work can replace those hashes with keyed or
salted evidence hashes if the evidence contract changes.
