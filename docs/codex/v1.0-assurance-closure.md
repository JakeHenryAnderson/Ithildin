# Ithildin v1.0 Assurance Closure

Status: v1.0 local-preview assurance closure map.

This document summarizes the release-candidate assurance posture for the v1.0 local-preview path. It
does not close external source-review rows by itself, approve capability expansion, or authorize
public/security-product positioning. It points reviewers to the gates that keep those claims honest.

## Assurance Questions

The v1.0 RC assurance pass should answer:

1. Are implemented local-preview lanes represented in the closure matrix?
2. Are critical/high findings absent for the current local-preview boundary?
3. Are accepted risks explicitly dispositioned and constrained?
4. Are external-pending rows still visible instead of being treated as closed?
5. Do release packets and README language avoid production/security/compliance claims?

## Current Assurance Ledger

| Evidence area | Command | RC interpretation |
| --- | --- | --- |
| Release guardrails | `make release-guardrails` | Public wording remains local-preview and non-production. |
| Finding records | `make reviewer-findings-check` and `make review-findings-summary` | Structured findings are valid and no critical/high findings are open. |
| Closure matrix | `make closure-matrix-evidence-sync` | Done milestone evidence is represented in the v3 closure matrix. |
| External closure honesty | `make external-review-closure-gate` | External closure is incomplete while pending rows remain visible. |
| Accepted risks | `make accepted-risk-register-check` | Local-preview risks are reviewed or accepted-deferred with blocked-claim implications. |
| Tool/power surface | `make tool-surface-invariant-gate` and `make no-new-powers-guardrail` | Tool count and deferred-power boundaries remain explicit. |
| Capability decisions | `make capability-decision-report` | Capability expansion and public/security-product positioning remain blocked. |
| Evidence confusion | `make evidence-confusion-gate` | Evidence wording does not confuse local evidence with custody-grade proof. |
| Redaction | `make packet-redaction-scan` | Review packets remain safe to hand off when findings are zero. |

## Closure State

- Local-preview implemented lanes may be treated as internally represented only when their matrix
  rows, evidence docs, gates, and source-review handoff artifacts agree.
- External/source review is not complete while `external-review-closure-gate` reports pending rows.
- Accepted-deferred risks are not closed risks. They remain constraints on product claims,
  capability design, and capability implementation.
- v1.0 RC assurance closure means the local-preview release packet is coherent and honest; it does
  not close external review and does not mean production approval, security-product approval,
  compliance approval, sandbox approval, or external custody.

## Stop Conditions

Stop v1.0 RC progress if:

- any critical/high finding is open;
- a document claims external review closure while external rows are pending;
- an accepted-deferred risk is described as resolved;
- a gate requires changing the product boundary;
- capability expansion, production identity, sandbox orchestration, SIEM custody, compliance
  automation, remote MCP, hosted telemetry, runtime Postgres, shell/Docker/Kubernetes/browser powers,
  arbitrary HTTP, broad writes, or plugin SDK behavior becomes necessary.

## Non-Goals

This assurance closure map does not prove implementation correctness, host compromise resistance,
OS isolation, production deployment safety, compliance automation, SIEM custody, production
identity, external notarization, or activity outside Ithildin-mediated actions.
