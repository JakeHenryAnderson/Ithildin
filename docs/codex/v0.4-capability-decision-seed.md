# v0.4 Capability Decision Seed

Task 151 records the decision seed for what happens after v0.4 review handoff.

## Default Decision

No new powerful tool class is approved by v0.4 completion alone.

Capability expansion remains blocked until external/source review addresses the high-risk local
preview surfaces and any critical/high findings are fixed or explicitly accepted as deferred.

## Powers That Remain Deferred

- shell execution;
- Docker socket access;
- Kubernetes tools;
- browser automation;
- arbitrary HTTP methods, caller-supplied headers, request bodies, cookies, or broad network access;
- broad filesystem writes, deletes, moves, chmod, archive extraction, or secrets-manager tools;
- plugin SDK or marketplace;
- remote hosted MCP gateway;
- production identity integrations;
- runtime Postgres;
- hosted telemetry collectors;
- managed model serving or LLM proxy workflows.

## Possible v0.5 Entry Criteria

Before planning a capability-expansion v0.5, require:

1. external/source review of patch apply, filesystem, HTTP, signed evidence, policy parity, MCP
   ingress, and review-console evidence;
2. no open critical/high findings;
3. closure matrix rows updated with reviewer, date, finding count, fixed commit or accepted risk,
   verification command, and closure state;
4. threat model updated if the proposed capability changes the runtime boundary;
5. user decision on whether v0.5 remains local-preview hardening or begins a carefully scoped new
   power class.

## Recommended Next Planning Question

Ask the external reviewer:

> Given the v0.4 evidence and source-review status, what is the safest next v0.5 objective, and what
> must remain blocked?
