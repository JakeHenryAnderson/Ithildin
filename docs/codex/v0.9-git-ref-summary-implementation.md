# v0.9 git.show.ref_summary Implementation

Status: approved v0.9 implementation for one read-only Git ref metadata capability.

`git.show.ref_summary` adds bounded read-only Git ref metadata through the existing governed
pipeline. It is a narrow continuation of the reviewed Git metadata lane, not a new powerful tool
class.

## Boundary

This implementation:

- adds one manifest: `tool-manifests/git-show-ref-summary.yaml`;
- keeps risk `read`, category `git`, and MCP exposure under the existing registry/listing path;
- uses fixed argv Git subprocess calls controlled by Ithildin;
- accepts only a structured selector: `all_local`, `branch`, or `tag`;
- accepts only a bounded `limit` plus optional trusted `workspace_id`;
- fails closed unless the Git repository toplevel is within or equal to the configured workspace
  root;
- returns response-local `ref_id` handles, ref kind, resolved commit hash, current-branch boolean,
  tag-peeling evidence, counts, and truncation evidence;
- returns no raw ref names and no stable ref-name hashes;
- skips refs that do not resolve to commits;
- fails closed on casefold-conflicting names, malformed Git metadata, oversized Git output,
  unsupported selector fields, or unsupported top-level fields;
- labels returned metadata as untrusted and records output-policy evidence;
- preserves policy preview/runtime parity with a syntactic `git_refs` resource;
- records safe audit metadata for selector kind, counts, truncation, and output-policy booleans;
- preserves existing policy rules, approval behavior, registry behavior, storage, identity, MCP
  transport, telemetry, and audit storage.

## Non-Goals

The tool provides no shell, no caller-supplied Git format strings, no arbitrary Git argv, no raw ref
names, no stable ref-name hashes, no remote refs, no refspecs, no pathspecs, no reflog access, no
checkout, no branch/tag mutation, no remote fetch, no credentials, no file contents, no raw diffs, no
blob contents, no symlink targets, no broad filesystem access, and no new write path.

## Privacy Caveat

`git.show.ref_summary` remains local-preview metadata mediation, not Git trust/provenance evidence
and not a Git sandbox. Response-local `ref_id` values are display handles only. They are not stable
identifiers, privacy-preserving randomness, or cross-response non-correlation guarantees. The privacy
boundary is that raw ref names and stable ref-name hashes are absent from responses and audit
evidence.
