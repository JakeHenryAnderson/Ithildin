# v0.4 Gating Overlay

This overlay records the hardened process added after GPT 5.5 Pro's roadmap/gating review. It is a
process contract for Tasks 123-151, not a new capability plan. It keeps v0.4 focused on review
closure, evidence maturity, diagnostics, and local-preview hardening.

The machine-readable source is [v0.4-milestone-manifest.json](v0.4-milestone-manifest.json).

## Required Per-Task Metadata

Every planned Task 123-151 must declare:

- `blocks_capability_expansion`: `yes`, `no`, or `conditional`;
- `blocks_broader_distribution`: `yes`, `no`, or `conditional`;
- `blocks_external_handoff`: `yes`, `no`, or `conditional`;
- `requires_external_review_before_closure`: boolean;
- `required_commands`: non-empty command list;
- `risk_surface`: non-empty list of affected trust surfaces;
- `deferred_boundary_must_remain_unchanged`: boolean.

This prevents vague roadmap items from quietly becoming product-boundary changes.

## Early Horizontal Gates

GPT 5.5 Pro's review called out that schema validation, packet diffing, guardrails, secret scanning,
and test determinism should happen before or alongside early v0.4 trust-surface work. v0.4 therefore
front-loads these tasks before deeper audit, policy, registry, MCP, UI, or executor assurance work:

- Task 123: v0.4 gating overlay;
- Task 124: release evidence schema gate v2;
- Task 125: review packet diff gate v2;
- Task 126: release guardrail expansion v2;
- Task 127: secrets hygiene and packet redaction scanner;
- Task 128: test isolation and determinism gate.

After every v0.4 wave, run:

```sh
make release-check
make review-candidate
make reviewer-findings-check
make review-run-manifest-check
make review-packet-diff OLD=<prior checkpoint> NEW=<current checkpoint>
git status --short
```

Also run focused tests for the touched subsystem before each checkpoint commit.

## Named Gates

### Capability Expansion Gate

Before planning or implementing any new powerful tool class, all of these must be true:

- `make release-check` passes;
- `make review-candidate` passes;
- `make reviewer-findings-check` passes;
- `make review-run-manifest-check` passes;
- `make review-packet-diff` passes against the prior checkpoint;
- `git status --short` is empty;
- no critical/high finding is open in the source-review closure matrix;
- relevant external-review rows are closed or explicitly accepted-deferred;
- accepted risks are linked in the threat model or release notes;
- tool-list changes are absent or covered by a boundary decision;
- deferred-power list changes are absent or covered by a boundary decision.

### Tool-Surface Invariant Gate

After every wave, assert:

- tool count remains 10;
- tool names are unchanged;
- no manifest additions occurred;
- no arbitrary HTTP method/header/body/cookie support was added;
- no broad filesystem write/delete/move/chmod/archive operation was added;
- no new MCP transport was added;
- no identity, storage, telemetry, or platform-support boundary changed.

### Evidence-Confusion Gate

After signed evidence, audit, release-evidence, and packet changes, assert:

- signed evidence is always described as local evidence;
- embedded public key versus trusted public key behavior is explicit;
- invalid audit chains can be exported only with invalid-status evidence;
- bundle type, key ID, algorithm, digest, and event order are tested;
- release evidence distinguishes runtime signing from demo signing;
- docs do not imply custody, notarization, immutable storage, or official release signing.

### UI/Admin No-Mutation Gate

After review-console/admin changes, assert:

- approval UI can approve/deny only through existing APIs;
- diagnostics remain read-only;
- incomplete apply-state display does not introduce repair mutation;
- copy-evidence actions do not create approvals or tool calls;
- unauthorized/failure states do not leak secrets or unnecessary local paths.

### External-Review Closure Gate

Before Task 151 is considered complete:

- every external finding has a structured finding file;
- every critical/high finding is fixed or formally accepted with rationale;
- every fix has a verification command;
- closure matrix state matches finding files;
- final review packet includes open accepted risks;
- no production-readiness or security-product wording appears.

## Boundary Rule

Task 151 creates a review packet. It does not unlock v0.5 or new powers by itself. A separate
capability-expansion decision must interpret external review results before any future tool-power
expansion.
