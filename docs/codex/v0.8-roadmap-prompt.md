# v0.8 Roadmap Prompt

Use this prompt when asking GPT 5.5 Pro / Very High or a human expert to help shape the next
roadmap after the v0.7 source-review closure work.

**Current status:** v0.8 roadmap/product-risk consultation after v0.6/v0.7 focused
source-review lane closure for the v0.1 local-preview runtime boundary. All eight focused v0.7
source-review lanes are closed for the v0.1 local-preview runtime boundary. The remaining open rows
are docs/product-decision rows, not unreviewed executor lanes. This does not approve
public/security-product positioning or new governed tool powers.

```text
You are advising on Ithildin's v0.8 roadmap as an expert external consultant.

Context:
- Ithildin is a local-first governed MCP/tool gateway for AI-agent tool use.
- The runtime boundary remains the narrow v0.1 local-preview boundary.
- The v0.6/v0.7 work focused on source-review closure, evidence maturity, and no-new-powers
  guardrails.
- Focused source-review lanes now closed for local-preview:
  - patch apply;
  - filesystem/platform;
  - http.fetch;
  - signed evidence, audit integrity, and manifest-lock verification;
  - policy/registry;
  - stdio MCP ingress;
  - review console/admin;
  - release/evidence automation.
- The closure gate is still intentionally not a public/security-product approval. It leaves 13
  docs/product-decision rows open: documentation IA, threat-model refresh, packet generators,
  roadmap/decision docs, capability gates, evidence-confusion gates, and boundary/handoff docs.
- Capability expansion remains blocked until v0.8 explicitly decides what can safely move next.

Please recommend a v0.8 roadmap that distinguishes:
1. cleanup required before public-preview positioning;
2. accepted-risk/product-decision work that should happen before any capability expansion;
3. the safest next implementation-hardening work;
4. whether any new governed tool class should remain deferred;
5. what evidence or external review should be required before changing the boundary.

Please do not approve or recommend immediate implementation of shell execution, Docker socket
access, Kubernetes tools, browser automation, arbitrary HTTP methods/headers/bodies/cookies, broad
network access, broad filesystem writes/deletes/moves/chmod/archive extraction, production identity,
runtime Postgres, hosted telemetry, remote MCP hosting, plugin SDK/marketplace, secrets-manager
tools, hosted model serving, or external notarization unless you also specify a concrete staged
review gate that must happen first.

Please answer in this structure:
- Overall judgment.
- What is genuinely closed now.
- What is still open or ambiguous.
- v0.8 goals, ordered by priority.
- Suggested v0.8 task waves.
- Gates/checks that should block each wave.
- What should remain deferred.
- What to send back for the next major external review.
- Brutal short version.
```
