# v0.2 Planning Seed

This is the starting plan for v0.2 after the v0.1 public-preview review. It should be reconciled
with external reviewer feedback before being treated as a committed roadmap.

## Planning Principle

v0.2 should strengthen trust semantics before adding new governed powers. The highest-value work is
not more tools; it is clearer evidence, stronger approval equivalence, better review workflows, and
more testable policy.

## Proposed Theme

**Trust Evidence v2:** make Ithildin easier to review, harder to misconfigure, and more precise
about what was approved, executed, and exported.

1. **Evidence Signing Sprint**
   - signed audit export format; done in Task 044;

   - verification commands and docs;

2. **Policy Confidence Sprint**

3. **Workspace and Approval UX Sprint**

4. **Local Operations Polish Sprint**

5. **Evidence Clarity Sprint**

   - no new tool powers.

## Keep Deferred

- Shell execution.
- Docker socket access.
- Kubernetes tools.
- Browser automation.
- Arbitrary HTTP methods, caller-supplied headers, request bodies, cookies, or broad network access.
- Broad filesystem writes, deletes, moves, chmod, archive extraction, or secrets-manager tools.
- Plugin SDK or marketplace.
- Remote hosted MCP gateway.
- Production identity integrations: OIDC, SAML, SCIM, hosted sessions, tenant stores.
- Runtime Postgres unless concurrency or multi-user requirements become unavoidable.
- Hosted telemetry collectors.
- Managed model serving or LLM proxy workflows.

## Review Questions Before Finalizing v0.2

- Did the external v0.1 review identify release blockers that should become v0.1 RC fixes instead of
  v0.2 tasks?
- Which follow-ups marked in the [v0.1 Security Test Matrix](v0.1-security-test-matrix.md) should
  become first-class v0.2 tasks?
- Which evidence fields are missing from approvals, audit events, exports, or UI views?
- Should OPA become canonical in v0.2, or remain optional evidence/prototype support?
- What is the minimal local admin-auth improvement that does not imply production identity?
- What explicit workspace semantics are needed before any additional write-like tool exists?
