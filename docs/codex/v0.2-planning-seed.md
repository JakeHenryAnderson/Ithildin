# v0.2 Planning Seed

This is the starting plan for v0.2 after the v0.1 public-preview review. It should be reconciled
with external reviewer feedback before being treated as a committed roadmap.

## Planning Principle

v0.2 should strengthen trust semantics before adding new governed powers. The highest-value work is
not more tools; it is clearer evidence, stronger approval equivalence, better review workflows, and
more testable policy.

## Proposed Theme

**Trust Evidence v2:** make Ithildin easier to review, harder to misconfigure, and more precise
about what was approved, executed, and exported.

## Candidate Tasks

| Priority | Task | Why now |
| --- | --- | --- |
| 1 | Signed audit exports | More trust value than runtime Postgres; lets users preserve local evidence outside mutable runtime state. |
| 2 | Signed manifest locks | Strengthens the trusted tool supply-chain boundary beyond local hash pinning. |
| 3 | Policy test harness | Lets maintainers and users assert expected allow/deny/approval decisions before policy changes ship. |
| 4 | Workspace model | Replaces a single root with named workspaces, explicit roots, and clearer per-workspace review context. |
| 5 | Approval review UX v2 | Show exact proposal/action hashes, base hash, manifest/policy evidence, requesting principal, expiry, stale status, and replay state. |
| 6 | Policy diff and impact preview | Make policy changes reviewable by showing affected tools, risks, principals, and sample decisions. |
| 7 | Local admin auth hardening | Improve token setup/rotation, local-only warnings, and admin session ergonomics without claiming production identity. |
| 8 | Audit chain repair/export diagnostics | Help users understand tamper or truncation failures without weakening fail-closed behavior. |
| 9 | Redaction evidence UX | Surface redaction counts/types and safe summaries without treating redaction as a security boundary. |
| 10 | MCP Inspector recipes | Add reproducible local recipes for tools/list, tools/call, approval-required, denial, and audit verification flows. |

## Candidate v0.2 Sprint Sequence

1. **Evidence Signing Sprint**
   - signed audit export format;
   - signed manifest lock format;
   - verification commands and docs;
   - no new tool powers.

2. **Policy Confidence Sprint**
   - policy test fixtures;
   - policy diff/impact preview;
   - stronger decision evidence snapshots;
   - YAML/OPA positioning cleanup.

3. **Workspace and Approval UX Sprint**
   - named workspace model;
   - approval review panel improvements;
   - stale/replay/evidence visualization;
   - no broad write expansion.

4. **Local Operations Polish Sprint**
   - admin token rotation/setup guidance;
   - MCP Inspector recipes;
   - audit diagnostics;
   - release docs refresh.

## Keep Deferred

- Shell execution.
- Docker socket access.
- Kubernetes tools.
- Browser automation.
- Arbitrary HTTP methods, caller-supplied headers, request bodies, cookies, or broad network access.
- Broad filesystem writes, deletes, moves, chmod, archive extraction, or secrets-manager tools.
- Plugin SDK or marketplace.
- Remote hosted MCP gateway.
- Production identity integrations: OIDC, SAML, SCIM, hosted sessions, tenant stores.
- Runtime Postgres unless concurrency or multi-user requirements become unavoidable.
- Hosted telemetry collectors.
- Managed model serving or LLM proxy workflows.

## Review Questions Before Finalizing v0.2

- Did the external v0.1 review identify release blockers that should become v0.1 RC fixes instead of
  v0.2 tasks?
- Which evidence fields are missing from approvals, audit events, exports, or UI views?
- Should OPA become canonical in v0.2, or remain optional evidence/prototype support?
- What is the minimal local admin-auth improvement that does not imply production identity?
- What explicit workspace semantics are needed before any additional write-like tool exists?
