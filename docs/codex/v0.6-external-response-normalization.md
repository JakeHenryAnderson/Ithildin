# v0.6 External Response Normalization

Task 184 adds a normalization step for raw GPT 5.5 Pro / Very High or human expert review
responses. The normalizer creates structured JSON finding drafts from review prose, but it does not
write finding records, close external-review rows, mark accepted risks reviewed, or approve capability
expansion.

## Command

```bash
uv run python scripts/external_response_normalize.py raw-response.md \
  --reviewer "GPT 5.5 Pro" \
  --reviewer-type "external-model" \
  --source-access "source-level" \
  --reviewed-commit "$(git rev-parse HEAD)" \
  --reviewed-packet-hash "sha256:<packet digest>" \
  --area "http-fetch" \
  --output var/review-runs/v0.6/http-fetch/normalized-response.json
```

## Required Metadata

- reviewer identity;
- reviewer type;
- source access level: `source-level`, `packet-and-source`, `packet-only`, or `docs-only`;
- reviewed commit;
- reviewed packet hash in `sha256:<64 lowercase hex>` form;
- reviewed area matching one of the focused v0.6 lane slugs.

## Finding Table Shape

The normalizer reads Markdown table rows and tab-separated rows using the existing v2 finding
extraction table shape:

| Finding ID | Severity | Area | Affected files/functions | Blocking status | Disposition | Recommended fix |
| --- | --- | --- | --- | --- | --- | --- |
| EXT-### or EXT-PA-### | critical/high/medium/low/informational | area | paths/functions | blocking/should-fix/later/advisory | open | fix summary |

Lane-specific namespaces such as `EXT-PA-###`, `EXT-FS-###`, `EXT-HTTP-###`, `EXT-SE-###`,
`EXT-PR-###`, `EXT-MCP-###`, `EXT-UI-###`, and `EXT-REL-###` are accepted for v0.6 focused review
lanes only when they match the requested `--area`. Generic `EXT-###` rows remain accepted when the
row `Area` matches the requested lane.

If a reviewer has no findings, the response must explicitly state `No findings` or equivalent. A
malformed response without valid finding rows does not normalize to a clean zero-finding result.

Source-level and packet-and-source reviews must name affected files/functions for each finding. If a
review is packet-only, that limitation stays visible in the normalized output and cannot by itself
close source rows.

## Safety Properties

- Secret-like markers are rejected.
- Missing reviewer metadata is rejected.
- Invalid severity, blocking status, disposition, commit, packet hash, area, or lane namespace is
  rejected.
- Output JSON records `mutates_findings: false` and `closes_external_review: false`.
- Critical/high findings still require later structured finding files, fixes or accepted-deferred
  rationale, and closure-matrix updates.
