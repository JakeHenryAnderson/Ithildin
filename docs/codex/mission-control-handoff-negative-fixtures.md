# Mission Control Handoff Negative Fixtures

Status: design-only negative fixture plan for the Mission Control handoff contract.

This document defines rejected or warning-required handoff payload shapes for any future Mission
Control display/import implementation. It does not add runtime behavior, API callbacks, MCP
transports, Mission Control execution behavior, local model invocation, VM/container lifecycle
management, sandbox orchestration, trusted-host promotion, SIEM adapters, production IAM, runtime
Postgres, hosted telemetry, shell, Docker/Kubernetes/browser governed powers, arbitrary HTTP, broad
filesystem writes, compliance automation, or public/security-product claims.

Validate the current negative fixture expectations with:

```sh
make mission-control-handoff-negative-fixtures-check
```

## Fixture Source

The positive seed payload is the current metadata-only handoff generated by:

```sh
make hello-world-mission-control-handoff
```

The negative fixtures mutate that seed in memory. They do not write runtime artifacts and do not
call Mission Control.

The concrete JSON fixture pack for future Mission Control importer tests is generated by:

```sh
make mission-control-handoff-fixture-pack
```

It writes one valid payload and the `MC-HANDOFF-NEG-001` through `MC-HANDOFF-NEG-014` negative
payloads under `var/review-packets/v3/mission-control-handoff-fixtures/`.

## Required Rejections

A future Mission Control display/import implementation must reject or visibly warn on these payload
families before displaying the handoff as usable mission evidence:

| Fixture ID | Mutation | Expected outcome |
| --- | --- | --- |
| `MC-HANDOFF-NEG-001` | missing `schema_version` | reject as unsupported schema |
| `MC-HANDOFF-NEG-002` | unsupported `schema_version` | reject as unsupported schema |
| `MC-HANDOFF-NEG-003` | `status` changed away from `metadata_only` | reject as live integration claim |
| `MC-HANDOFF-NEG-004` | `mission_control_runtime_behavior: true` | reject as execution overclaim |
| `MC-HANDOFF-NEG-005` | `host_promotion_performed: true` | reject as promotion overclaim |
| `MC-HANDOFF-NEG-006` | `ithildin_remains_policy_authority: false` | reject as policy-authority drift |
| `MC-HANDOFF-NEG-007` | absolute attachment path | reject as unsafe packet link |
| `MC-HANDOFF-NEG-008` | parent-directory traversal attachment path | reject as unsafe packet link |
| `MC-HANDOFF-NEG-009` | missing `display_contract` | reject as missing display rules |
| `MC-HANDOFF-NEG-010` | hidden-field denylist missing `tokens` | reject as unsafe display contract |
| `MC-HANDOFF-NEG-011` | warning chips missing `host_promotion_not_performed` | reject as missing warning state |
| `MC-HANDOFF-NEG-012` | `mission_control_authority` claims executor authority | reject as authority overclaim |
| `MC-HANDOFF-NEG-013` | payload includes raw file contents | reject as content leak |
| `MC-HANDOFF-NEG-014` | payload includes raw prompt text | reject as prompt leak |

## Safe Error Expectations

Negative fixture validation should report stable reason labels only. It must not echo file contents,
raw prompts, raw host paths, environment values, tokens, private keys, package script values, or
response bodies.

## Current Implementation Boundary

The fixture plan is an Ithildin-side contract and validation aid. It does not implement a Mission
Control importer, live display behavior, bidirectional API, callback path, approval path, audit
authority transfer, sandbox orchestration, VM/container lifecycle management, or trusted-host
promotion.
