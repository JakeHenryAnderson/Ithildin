## Future Runtime Shape

If a later decision record approves implementation, the only future runtime shape this contract
allows planning for is:

```text
sandbox://artifact -> host-staging://artifact -> approved://artifact
```

The future implementation must be one artifact, one approval, one promotion attempt, and one bounded destination label.
Runtime planning must keep raw filesystem paths out of operator-facing evidence and must model labels
as evidence identifiers until a separately reviewed storage resolver binds them to exact safe
locations.

Any future tool or endpoint proposal must be reviewed as a new power-bearing surface. This contract
does not approve a manifest, executor, policy rule, API route, MCP exposure, UI action, Mission
Control action, local model action, sandbox orchestration action, or SIEM adapter.

## Promotion Request Contract

A future trusted-host promotion request must be represented as safe metadata before any runtime
implementation can be proposed. The future request shape must be closed and must include only:

| Field | Purpose | Boundary |
| --- | --- | --- |
| `promotion_request_id` | Stable request identifier | Generated by Ithildin; no caller-supplied filesystem meaning |
| `workspace_id` | Workspace binding | Must match the governed workspace registry |
| `sandbox_descriptor_id` | Operator-reviewed sandbox/host posture binding | Descriptor evidence only; no lifecycle control |
| `source_zone_label` | Source label such as `sandbox://artifact` | Must use an allowed zone prefix, not a raw path |
| `destination_zone_label` | Destination label such as `host-staging://artifact` | Must use an allowed zone prefix, not a raw path |
| `artifact_id` | Response-local artifact identifier | Must not encode file names, host paths, customer names, or secrets |
| `artifact_sha256` | Exact source artifact digest | Required before approval and rechecked before placement |
| `artifact_size_bytes` | Bounded size evidence | Must be inside the future limit before review |
| `artifact_media_label` | Coarse allowlisted type label | Labels only, such as `text`, `json`, `markdown`, or `unknown_safe_text` |
| `requested_by_principal` | Principal that requested review | Must come from the principal registry, not caller-supplied roles |
| `requested_at` | Timestamp evidence | Informational; not a trust root |
| `output_policy` | Secret-free disclosure policy | Must state that file contents, raw paths, diffs, prompts, responses, and secrets are excluded |

The future request must reject unknown fields, caller-supplied host paths, URL-shaped labels, absolute
paths, parent traversal, encoded traversal, hidden/sensitive path labels, control characters, Unicode
normalization ambiguity, symlink or hardlink claims, directory promotion, archive extraction, and
unbounded artifacts. A future request may not contain file contents, prompt text, response bodies,
diffs, package script values, dependency names, environment names or values, registry URLs, private
keys, shell output, VM logs, raw sandbox-internal paths, or stack traces.

## Zone And Artifact Identity Contract

The allowed future zone model is evidence-first:

- `sandbox://` labels identify an operator-reviewed sandbox artifact source.
- `host-staging://` labels identify a bounded host staging destination.
- `approved://` labels identify a later approved evidence state, not broad host authority.
- `evidence://` labels identify audit/export evidence related to the request.

A future implementation plan must define a deterministic normalized label form before hashing or
approval. Normalization must fail closed on empty labels, repeated separators, trailing ambiguity,
casefold ambiguity, percent-encoding ambiguity, control characters, Unicode ambiguity, path
traversal, URL schemes outside the four allowed label prefixes, raw filesystem paths, and labels
that imply overwrite, delete, move, chmod, recursive copy, or archive extraction.

Artifact identity must bind:

- `artifact_id`;
- `source_zone_label`;
- `destination_zone_label`;
- `workspace_id`;
- `sandbox_descriptor_id`;
- `artifact_sha256`;
- `artifact_size_bytes`;
- `artifact_media_label`;
- `proposal_hash`;
- `request_hash`;
- `expected_staging_sha256`;
- `expected_approved_sha256` if an approved-zone transition is ever separately proposed.

The first future runtime slice should plan for staging only. Any `approved://` transition must stay
design-only unless a later decision record explicitly approves it with a separate one-time approval
scope.

## Approval And Evidence Binding Plan

Future trusted-host promotion must use stored-proposal-only semantics. A runtime call may consume
only a previously stored promotion proposal and a one-time approval whose immutable scope exactly
matches that proposal.

A future approval scope must bind:

- `tool_name`;
- `promotion_request_id`;
- `promotion_proposal_id`;
- `proposal_hash`;
- `artifact_id`;
- `artifact_sha256`;
- `artifact_size_bytes`;
- `source_zone_label`;
- `destination_zone_label`;
- `workspace_id`;
- `sandbox_descriptor_id`;
- `trusted_host_descriptor_hash`;
- `manifest_hash` and manifest version;
- `policy_engine`, policy hash, policy version, and policy document version;
- matched policy rules and obligation keys;
- requesting principal and approving principal;
- request hash;
- one-time scope hash;
- approval expiry;
- schema/tool input version;
- reviewed packet commit or implementation gate commit.

Before any future placement attempt, the executor must revalidate the current proposal hash,
artifact hash, destination label, workspace binding, sandbox descriptor hash, manifest hash/version,
policy hash/version when required, schema/tool input version, requesting principal, approval expiry,
approval status, and one-time scope hash. Mismatch, expiry, replay, stale artifact hash, changed
destination, wrong tool, wrong principal, missing descriptor, disabled workspace, or missing policy
evidence must fail closed before placement.

Approval consumption must be atomic and one-time. The future state machine must use compare-and-set
transitions so concurrent calls cannot both consume the same approval or create more than one
promotion attempt for a single approval ID.

## Attempt, Diagnostics, And Audit Plan

A future attempt store must record safe metadata only:

- `promotion_attempt_id`;
- approval ID;
- proposal ID;
- request ID;
- workspace ID;
- sandbox descriptor ID;
- source and destination zone labels;
- artifact hash and expected post-placement hash;
- phase/status;
- failure reason label;
- timestamps;
- safe counts and warning labels.

The future attempt store must never record file contents, raw host paths, raw sandbox paths, diffs,
prompts, model responses, secrets, VM logs, stack traces, dependency names, package scripts,
environment names or values, registry URLs, or private keys.

Diagnostic behavior must be read-only unless a separate decision record approves mutating
reconciliation. A future diagnostic view may report `clean`, `recovery_required`, `ambiguous`, or
`blocked` with safe reason labels and operator recommendations. It must not repair, retry, roll
back, complete, delete, overwrite, or move artifacts.

Audit evidence must include only labels, hashes, IDs, policy/manifest evidence, approval IDs,
decision labels, state labels, redaction status, and safe counts. Audit events must not echo
artifact contents or raw filesystem paths.

## Required Future Components

A future implementation plan must define:

- an exact artifact hash-binding model for source artifact, host staging artifact, and approved
  host artifact;
- an approval binding model for approval ID, request hash, one-time scope hash, approval expiry,
  source label, staging label, approved label, policy hash, manifest hash, operator principal,
  workspace ID, sandbox ID, schema/tool version, and reviewed packet commit;
- an attempt store with compare-and-set state transitions and one attempt per approval ID;
- a read-only diagnostic model for incomplete or recovery-required attempts;
- a safe storage resolver that rejects arbitrary host paths, hidden paths, `.git`, symlinks,
  hardlinks, directories, unsupported file types, binary targets, archive extraction, absolute paths,
  parent traversal, encoded traversal, URL-shaped labels, Unicode ambiguity, and control characters;
- an atomic placement model that cannot overwrite, delete, move, chmod, merge directories, or recursively copy;
- negative transcripts for conflict, replay, stale evidence, unsafe labels, path escape, sensitive
  payloads, and product-boundary overclaims;
- audit evidence limited to labels, hashes, IDs, state labels, decision labels, and safe counts;
- rollback/reconciliation non-goals unless a separate decision record approves a mutating recovery
  design.
