# v0.4 Boundary Charter

Decision date: 2026-05-31

v0.4 is a review-closure and evidence-maturity wave for the v0.1 local-preview runtime boundary. It
is not a capability-expansion release.

## Allowed Work

v0.4 may change tests, evidence contracts, review automation, diagnostics, hardening behavior, UI
clarity, documentation, and local-preview release machinery. It may strengthen existing executors
and evidence workflows when doing so preserves the current tool-power boundary.

## Deferred Work

Do not add:

- shell execution;
- Docker socket access;
- Kubernetes tools;
- browser automation;
- arbitrary HTTP methods, caller-supplied headers, request bodies, cookies, or broad network access;
- broad filesystem writes, deletes, moves, chmod, archive extraction, or secrets-manager tools;
- plugin SDK or marketplace;
- remote hosted MCP gateway or network MCP transport;
- production identity integrations, OIDC, SAML, SCIM, tenants, or hosted sessions;
- runtime Postgres;
- hosted telemetry collectors;
- managed model serving or LLM proxy workflows;
- external audit anchoring, hosted notarization, official supply-chain signing, or custody-grade
  evidence.

## Fixed Boundaries

- SQLite remains the only runtime storage backend.
- Stdio-only local MCP remains the transport boundary.
- Local principals remain local attribution labels, not production identity.
- Local signed evidence remains local operator evidence, not external notarization.
- macOS and Linux are the only filesystem security-supported local-preview profiles; Windows and
  WSL remain unsupported for workspace/race security claims unless separately reviewed.

## Stop Conditions

Stop autonomous implementation and report status if:

- a critical or high internal/external finding is open;
- source behavior contradicts the threat model, evidence contracts, or this charter;
- the implementation requires changing the deferred-power list;
- the same release or trust-boundary test fails three times for the same suspected reason;
- the working tree cannot be returned to a clean verified state;
- an external reviewer or user asks for a boundary decision before more implementation.

## v0.4 Success Criteria

v0.4 succeeds when review runs, findings, closure state, adversarial hardening evidence, release
evidence, and review packets are reproducible and clearly separated from production/security-product
claims. It does not succeed by adding new tool powers.
