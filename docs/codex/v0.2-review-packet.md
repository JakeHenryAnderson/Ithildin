# Ithildin v0.2 Review Packet

This packet is the starting point for an external/code review of Ithildin after the v0.2
trust-evidence wave. It is meant to help a reviewer assess whether the local-preview boundary,
evidence model, approval binding, executor limits, and documentation claims match the implementation.

## Review Scope

Review Ithildin as a local-first governed MCP/tool gateway. The core question is whether it is a
credible local mediation layer for narrow AI-agent tool use, with honest limits and useful evidence.

This is not a request to expand capabilities. New shell, Docker socket, Kubernetes, browser
automation, broad filesystem write, remote MCP, plugin marketplace, production identity, runtime
Postgres, and hosted telemetry work remain out of scope for this review.

## Current Boundary

Ithildin assumes the local host, local admin, trusted tool manifests, principal registry, workspace
registry, and policy files are part of the trusted computing base. The agent/model is untrusted and
interacts through stdio MCP or admin-mediated local APIs.

Audit records are local tamper-evident evidence. Signed exports and signed manifest locks improve
offline integrity checks, but they are not external notarization or hosted custody.

## Implemented Capability Snapshot

- Stdio MCP adapter backed by the same governed pipeline as the API.
- Trusted YAML tool manifests pinned by `tool-manifests.lock.json`.
- Optional local Ed25519 signatures for audit exports and manifest locks.
- Local principal registry with role-aware tool visibility.
- Named local workspaces for scoped read, git, and patch proposal tools.
- Deny-default YAML policy with committed offline policy fixtures.
- Policy preview, policy impact preview, and runtime policy decision evidence.
- Approval-gated stored patch application with proposal, base, manifest, policy, schema, principal,
  and request evidence checks.
- Review console for system trust, tool registry, approvals, patches, policy preview/impact, audit,
  diagnostics, and export actions.
- Read-only filesystem and git tools scoped to named workspaces.
- Stored patch proposal and approval-gated patch apply as the only write path.
- GET-only exact-allowlist `http.fetch` with redirect, DNS/IP, canonical-host, proxy, timeout, and
  response-size regression coverage.

## Evidence Commands

Run these from the Ithildin repo root before review handoff:

```sh
make release-check
make release-evidence
make release-packet
make docs-site
```

`make release-check` is the required gate. `make release-evidence` and `make release-packet` print
secret-free snapshots that include repo root, commit, dirty state, manifest lock status, policy
status, tool count, principal/workspace status, storage/telemetry posture, security boundaries, and
audit verification state.

## Reviewer Focus

- Does the documentation avoid overclaiming local mediation as OS isolation?
- Are the signed local evidence formats useful and clearly bounded?
- Are audit export signatures and manifest-lock signatures verifiable offline without implying
  hosted trust?
- Are policy preview, policy fixtures, impact preview, and runtime decision evidence comparable
  enough for review?
- Are approval binding checks strong enough for stored patch apply?
- Are replay, expiry, stale base, manifest drift, policy drift, and principal drift covered?
- Are filesystem path, symlink, hardlink, Unicode, `.git`, hidden path, binary, encoding, and atomic
  apply cases covered at the right level?
- Are HTTP redirect, DNS/IP, canonical host, IDNA, proxy inheritance, timeout, size, and safe-error
  cases covered at the right level?
- Are the review console surfaces showing enough evidence for local administrators to understand
  approvals and audit state?
- What must be resolved before any broader tool powers are considered?

## Deferred Boundaries

These remain deliberately deferred:

- Production authentication, OIDC, SAML, SCIM, hosted sessions, tenant stores, and managed identity.
- Runtime Postgres storage and production migrations.
- Hosted telemetry collectors and production observability dashboards.
- Remote MCP hosting or network MCP transport.
- Shell execution.
- Docker socket access.
- Kubernetes tools.
- Browser automation.
- Arbitrary HTTP methods, caller-supplied headers, request bodies, cookies, or broad network access.
- Broad filesystem writes, deletes, moves, chmod, archive extraction, or secrets-manager tools.
- Plugin SDK or marketplace.
- Managed model serving or LLM proxy workflows.
- External audit anchoring, official hosted supply-chain signing, or custody-grade evidence.

## Suggested Review Output

Please distinguish documentation risk from implementation risk. For implementation findings, include
file/function references when possible. For product findings, call out whether the issue blocks a
local-preview tag, should be fixed before broader distribution, or belongs in a later roadmap.
