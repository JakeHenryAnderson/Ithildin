# Ithildin v1.0 RC Feature Freeze

Status: v1.0 local-preview feature-freeze gate.

This document records the v1.0 RC feature freeze. It does not add runtime behavior, tool
manifests, executors, policy rules, API/MCP behavior, UI runtime behavior, sandbox orchestration,
SIEM adapters, production identity, runtime Postgres, hosted telemetry, remote MCP, plugin SDK
behavior, compliance automation, or new governed tool powers.

## Freeze Decision

Ithildin is frozen for v1.0 local-preview RC assurance.

- governed tool count remains `24`;
- latest implemented governed tool remains `sandbox.artifact.write_text`;
- no next capability is selected;
- capability expansion remains blocked;
- public/security-product positioning remains blocked;
- production/security/compliance positioning remains blocked;
- runtime changes are allowed only for blocking release fixes, test/gate fixes, documentation
  corrections, packet/evidence refreshes, and source-review finding remediation that does not
  change the product boundary.

## Allowed During Freeze

- release-check, review-candidate, packet, docs-site, and redaction fixes;
- source-review finding documentation and remediation for existing local-preview behavior;
- operator quickstart, demo, workbench, evidence, and handoff clarity;
- negative transcripts, fixture checks, and review packet regeneration;
- dependency-free validation scripts that enforce the existing boundary;
- UI/readiness tests that verify existing local-preview behavior.

## Blocked During Freeze

- new tool manifests;
- new executors;
- new policy powers;
- new MCP/API behavior;
- new UI runtime controls;
- new write, network, shell, Docker, Kubernetes, browser, sandbox orchestration, SIEM adapter,
  production identity, runtime Postgres, hosted telemetry, remote MCP, plugin SDK, compliance
  automation, or trusted-host promotion behavior;
- broad filesystem writes, deletes, moves, chmod, archive extraction, or unbounded traversal;
- claims of production readiness, sandbox safety, SIEM custody, compliance automation, immutable
  audit, or public/security-product suitability.

## Unfreeze Conditions

Feature work may resume only after a new explicit post-RC decision records:

- the proposed capability or architecture lane;
- why it is necessary before or after v1.0 RC;
- proposal, implementation plan, implementation gate, source-review handoff, policy/audit evidence,
  negative transcripts, and no-new-powers impact;
- updated tool-surface and release/readiness evidence;
- external/source review requirement where the change touches security, identity, storage,
  sandbox, network, write, or public-positioning claims.

## Required Gate

Run:

```bash
make v1-rc-feature-freeze
```

The gate verifies the current tool surface, no-new-powers posture, next-capability state,
capability-decision report, v1.0 RC status, v1.0 RC roadmap, and v1.0 RC readiness evidence.

## Stop Conditions

Stop and reassess if:

- a future sprint requires changing the deferred-power list;
- a critical/high trust-boundary finding opens;
- release work needs a new governed tool or new authority surface;
- Mission Control, sandbox, SIEM, identity, storage, or compliance wording implies current
  production readiness;
- a low/mini worker proposes runtime or trust-boundary changes.
