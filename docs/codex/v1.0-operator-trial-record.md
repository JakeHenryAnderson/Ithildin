# Ithildin v1.0 Operator Trial Record

Status: generated local-preview operator trial evidence record for the v1.0 RC path.

This document describes the generated operator trial record. It does not add runtime behavior,
tool manifests, executors, policy rules, API/MCP behavior, UI runtime behavior, sandbox
orchestration, SIEM adapters, production identity, runtime Postgres, hosted telemetry, remote MCP,
plugin SDK behavior, compliance automation, or new governed tool powers.

## Purpose

The operator trial checklist defines what an operator should run. The operator trial record captures
the current checked posture and handoff artifact pointers after those gates have been generated.

Generate the record with:

```sh
make v1-operator-trial-record
```

The generated output is ignored local evidence under:

```text
var/review-packets/v1.0/operator-trial/
```

Validate the record workflow with:

```sh
make v1-operator-trial-record-check
```

## Captured Evidence

The record is secret-free and records:

- current Git commit and dirty state;
- governed tool count;
- latest implemented tool;
- selected next capability;
- v1.0 RC status check result;
- v1.0 progress assessment result;
- operator trial checklist check result;
- v1.0 readiness check result;
- workbench readiness result;
- demo evidence readiness result;
- packet redaction scan status when review packets exist;
- generated artifact paths, existence, byte counts, and SHA-256 digests;
- safe recommended next commands.

## What The Record Does

- Provides a compact handoff note that the checklist-style evidence was generated for the current
  repository state.
- Points reviewers to the current v1.0 RC packet, operator workbench packet, demo evidence packet,
  live-demo packet, and release-check transcript when present.
- Keeps the distinction between checked local-preview evidence and external/source-review closure.

## What The Record Does Not Do

The record does not:

- start services;
- run Compose;
- call governed tools;
- approve actions;
- mutate workspaces;
- manage VMs or containers;
- invoke local models;
- transfer authority to Mission Control;
- promote artifacts to the trusted host;
- create SIEM custody;
- certify compliance;
- approve production identity;
- approve public/security-product positioning.

## Pass Interpretation

The record is useful when:

- `make v1-operator-trial-record-check` reports `valid: true`;
- `make release-check` passes;
- `make review-candidate` passes;
- packet redaction scan reports `findings: 0`;
- `git status --short` is empty after committed source changes and ignored packet generation.

This is still local-preview handoff evidence only. It is not production deployment readiness, a
kernel sandbox claim, compliance automation, SIEM-grade custody, external notarization, or proof of
activity outside Ithildin-mediated actions.
