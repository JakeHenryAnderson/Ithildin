# v0.3 External Review Prompt

Use this prompt with GPT 5.5 Pro / Very High or a human expert reviewer after providing the v0.3
review packet, release evidence, command outputs, consolidated packet, and relevant source files.

```text
You are acting as an expert security/product consultant reviewing Ithildin as a v0.3-prep source
review candidate for the v0.1 local-preview runtime boundary.

Review from the supplied packet, release evidence, source files, tests, and linked docs. If you
cannot inspect code directly, state that limitation and distinguish documentation risk from
implementation risk.

Project summary:
Ithildin is a local-first governed MCP/tool gateway for AI agents. It mediates local agent tool use
through trusted tool manifests, JSON Schema validation, local principal labels, named workspaces,
deny-default policy, approval, scoped execution, redaction, and local tamper-evident audit. The
v0.3-prep track adds assurance automation, adversarial tests, fail-closed validation suites,
evidence schema validation, packet diffing, release guardrails, and internal review prompts. It does
not add new tool powers and remains local preview only.

Review objective:
Determine whether Ithildin is ready to proceed from v0.3-prep into external/source review closure
for the local-preview boundary, and identify blockers before any broader tool powers or public
security-product positioning are considered.

Please review:

1. Boundary honesty
   - Does the project avoid claiming OS isolation, production security, immutable audit, hosted MCP,
     production identity, runtime Postgres, or broad arbitrary tool safety?

2. Patch apply
   - Inspect approval binding, proposal/base/manifest/policy/schema/principal/request/expiry
     evidence, apply attempt state, recovery diagnostics, stale-base rejection, and replay denial.

3. Filesystem and git
   - Inspect workspace confinement, symlink/hardlink denial, hidden/sensitive path denial, UTF-8 and
     size limits, git read-only command shapes, and macOS/Linux support assumptions.

4. HTTP fetch
   - Inspect exact allowlist matching, URL canonicalization, DNS/IP checks, redirect revalidation,
     pinned transport, proxy suppression, timeout/body limits, and safe errors. Confirm SUB-001 is
     remediated or identify remaining connect-time DNS/SSRF concerns.

5. Audit and signed evidence
   - Inspect SQLite/JSONL chain verification, exported JSONL verification, signed audit export
     verification, signed manifest lock verification, replay/substitution tests, and local-only
     trust-root wording.

6. Trusted configuration
   - Inspect manifest, manifest-lock, principal registry, and workspace registry fail-closed
     behavior and negative tests.

7. Policy and MCP
   - Inspect policy preview/runtime parity, YAML canonical policy stance, OPA sidecar boundary, MCP
     ingress thinness, fixed principal behavior, and no-bypass audit evidence.

8. Review console
   - Inspect whether approval binding evidence, scope hash, policy reason, diagnostics, audit state,
     and trust warnings are visible enough for a local admin.

9. Release automation
   - Inspect release evidence schema, release guardrails, review packet bundle hashes, consolidated
     attachment hashes, review packet diff command, negative transcripts, and internal review packet
     v2.

Output format:

- Overall judgment: ready / ready with fixes / not ready.
- Blocking findings: include severity, file/function references, and whether they block external
  closure or broader distribution.
- Should-fix before broader distribution.
- Documentation/positioning risks.
- Technical hardening priorities.
- Release packet gaps.
- Recommended next 5-10 tasks.
- Do-not-add-yet list.
- Brutal short version.
```
