# v0.2 External Review Prompt

Use this prompt with GPT 5.5 Pro / Very High or a human expert reviewer after providing the
[v0.2 Review Response and RC Cleanup](v0.2-review-response-and-rc-cleanup.md),
[v0.2 Review Packet](v0.2-review-packet.md), [Reviewer Reproduction Map](reviewer-reproduction-map.md),
[Source Review Closure Matrix](source-review-closure-matrix.md),
[Reviewer Finding Template](reviewer-finding-template.md), fresh release evidence, and linked
documents.

```text
You are acting as an expert security/product consultant reviewing Ithildin as a v0.2 review
candidate for the v0.1 local-preview runtime boundary. Review from the supplied packet, release
evidence, source files, and linked docs. If you cannot inspect code directly, state that limitation
and distinguish documentation risk from implementation risk.

Project summary:
Ithildin is a local-first governed MCP/tool gateway for AI agents. It mediates local agent tool use
through trusted tool manifests, JSON Schema validation, local principal labels, named workspaces,
deny-default policy, approval, scoped execution, redaction, and local tamper-evident audit. v0.2
adds trust-evidence hardening such as locally signed audit exports, locally signed manifest locks,
policy fixtures, policy impact preview, decision evidence, approval review evidence, audit
diagnostics, and security regression coverage. It remains a local preview, not production security
software.

Review objective:
Identify what should block a v0.2 review candidate for the v0.1 local-preview runtime boundary,
what should be fixed before broader public distribution, and what should be prioritized before any
new governed tool powers. Do not brainstorm new powerful tool capabilities unless they are framed as
deferred risks.

Please review these areas:

1. Boundary honesty and release posture
   - Does Ithildin accurately describe itself as local mediation rather than OS isolation?
   - Are local host trust, local admin trust, trusted manifests, trusted policy files, trusted
     workspaces, local principal labels, and audit limitations clear?
   - Are any docs, UI labels, release packet fields, or command names overclaiming trust?

2. Locally signed evidence and release packet completeness
   - Review signed audit exports and signed manifest locks as local operator evidence.
   - Review the non-production signed-evidence demo summary if it is attached.
   - Do the docs and evidence contracts avoid implying external notarization, hosted custody, or
     official supply-chain signing?
   - Is the release packet sufficient for an external reviewer to understand repo root, commit,
     dirty state, release-check transcript status, generated artifact hashes,
     policy/tool/principal/workspace status, audit state, and deferred boundaries?

3. Approval and patch-apply binding
   - Review stored-proposal-only patch apply and one-time approval consumption.
   - Check proposal hash, base file hash, manifest hash/version, policy evidence, schema hash,
     matched rules, principal, request hash, expiry, stale-base checks, drift checks, and replay
     prevention.
   - Are the review console evidence surfaces enough for a local admin to understand what is being
     approved?

4. Policy confidence and explainability
   - Review committed policy fixtures, policy test harness, policy impact preview, policy preview,
     and runtime decision evidence.
   - Are policy preview and runtime decisions comparable enough?
   - Are YAML policy and optional OPA evidence positioned safely?

5. Filesystem, workspace, git, and patch executor edge cases
   - Review named workspace scoping, traversal blocking, hidden/sensitive path denial, `.git`
     internals, symlink/hardlink behavior, Unicode/path ambiguity, binary/encoding rejection,
     stale-base rejection, and atomic patch apply.
   - What local-preview bypasses or race cases remain highest risk?

6. HTTP fetch edge cases
   - Review exact allowlist matching, GET-only behavior, no caller headers, unsupported scheme and
     credential rejection, redirects, redirect-to-private, DNS/IP checks, DNS changes, IPv6,
     encoded IPs, IDNA/punycode, proxy environment variables, timeouts, body-size limits, and safe
     error behavior.
   - What SSRF or normalization risks remain?

7. Audit, redaction, and diagnostics
   - Is audit correctly positioned as local tamper-evident evidence?
   - Are audit diagnostics useful without mutating evidence?
   - Are redaction summaries clearly best-effort and not treated as a security boundary?
   - Are signed exports, JSONL exports, and verification commands understandable?

8. Negative recipes and denial evidence
   - Review path traversal, symlink escape, stale-base patch apply, private redirect, unknown or
     disabled principal, and replayed approval recipes.
   - Are the expected denials clear enough for reviewer reproduction?

9. v0.3 or post-review roadmap
   - What must be fixed before adding any broader tool powers?
   - Which trust improvements should come before production identity, runtime Postgres, remote MCP,
     hosted telemetry, plugin SDKs, or new executors?

10. Source review closure
   - For each reviewed area, update the source review closure matrix with reviewer, date, files or
     functions inspected, finding count, blocking finding status, disposition, and closure notes.

Output format:

- Overall judgment: ready / ready with fixes / not ready.
- Blockers: ordered list with severity and rationale.
- Should-fix before broader distribution: concrete docs, tests, or implementation changes.
- Documentation and positioning risks: exact wording fixes where useful.
- Technical hardening priorities: concrete tests or implementation changes.
- Release packet gaps: anything missing from the evidence bundle or handoff docs.
- v0.3 priority roadmap: ordered tasks with why-now rationale.
- Do-not-add-yet list: dangerous expansions to keep deferred.
- Brutal short version: 5-8 sentences.
```
